## Problem Statement

Ambient music listeners and space enthusiasts consume cosmic-themed audio through passive, disconnected experiences — Spotify playlists, YouTube mixes, apps like Endel or Brain.fm. The visual and audio dimensions of the "space experience" exist in separate products. No product lets users navigate through a realistic solar system where the act of moving IS the act of composing music. The intersection — navigation-driven spatial audio in an immersive solar system — is an empty product category.

## Demand Evidence

The founder is the target user ("I'm the user"). No external demand validation yet. This is a taste-risk product, not a demand-risk product. The key validation question is: "does the prototype make ME feel what I imagined?" If yes, distribution and monetization become downstream problems.

Supporting signals: ambient music is a growing category (Endel raised $60M+, Brain.fm profitable, myNoise has loyal community). Space visualization apps have millions of downloads (Solar Walk, Solar System Scope). The intersection is unoccupied.

## Status Quo

Current workaround: ambient music listeners and space nerds exploring cosmic-themed mixes on streaming platforms. Audio and visuals completely disconnected. NASA's Space Jam does sonification but is educational, not meditative. Soundescape does 3D generative audio without space exploration. Solar Walk and NASA Eyes treat audio as background noise. The status quo is "play ambient playlist + look at space stuff separately."

## Target User & Narrowest Wedge

**Target user (hypothesis):** Someone in r/ambientmusic, r/space, r/meditation who uses apps like Endel, Calm, or Brain.fm. Has a fascination with space. Reaches for this at 10pm winding down or during deep work. Wants agency over mood without decision fatigue — navigation IS the decision.

**Narrowest wedge:** A web-based solar system where audio stems fade in/out by proximity. Prove the mechanic, then scale.

### Body Count Roadmap
- **Phase 1:** Sun + 8 planets (9 bodies) — prove the core audio-navigation coupling
- **Phase 2:** Add major moons + dwarf planets (~30 bodies) — test audio density and shared stem pools
- **Phase 3:** Remaining moons, asteroid belt region (~50 bodies) — full experience

## Constraints

- Zero paid dependencies — entire stack is open-source/public-domain (no Unity, no paid SDKs, no commercial APIs)
- Lightweight mobile app — small install, assets streamed on demand
- Web as flagship — self-hostable static build (no vendor lock-in), with managed hosting (Vercel/Netlify) as a convenience option
- No cloud AI — all computation client-side (procedural audio = Web Audio API oscillators/filters, not ML-based generation)
- Hybrid audio — composed stems for bodies, procedural ambient for deep space (Web Audio API oscillator/filter chains + short looping samples)
- Visual fidelity is phased — basic visuals first (NASA 2K textures, bloom/glow), photorealism layered in later

## Premises

1. The coupling of navigation and audio IS the product — neither visual nor audio alone is differentiated.
2. Visual fidelity is a phased investment. Prototype with basic visuals to prove the audio-navigation coupling. Layer photorealism later.
3. ~50 bodies is the starting target, adjust in practice. Not all need unique stems; some may share audio pools or be visual-only.
4. Mobile matters from day one (Capacitor wrapper around web engine).
5. Hybrid audio: composed stems for bodies, procedural ambient for deep space transitions (never silence).
6. Zero paid dependencies is a hard constraint.

## Approaches Considered

### Approach A: "Prove the Feeling" (MVP-first)
8-10 bodies, basic visuals, polished audio coupling. Web only. 1-2 weekends. Fastest path to experiencing the product. Doesn't prove mobile viability or test audio overload at scale.

### Approach B: "Full Blueprint" (Architecture-first) — CHOSEN
Design complete system architecture up front. Audio engine, asset pipeline, Capacitor mobile wrapper, streaming infrastructure. Build modular from day one so scaling to 50 bodies and photorealism is content, not engineering. 3-4 weeks to first playable. No throwaway code.

### Approach C: "Audio-First Prototype" (Lateral)
Skip 3D, build audio engine in 2D (flat solar system map, drag cursor, hear stems). Prove mixing and distance curves in 2D, then wrap in 3D scene. Isolates audio risk but misses the 3D spatial perception and "whoa" factor.

## Recommended Approach

**Approach B: Full Blueprint.** Build the architecture right from the start. The key modules:

### 1. Rendering Engine (Three.js)
- Scene graph with Sun at origin, planets at ephemeris-computed positions
- NASA 2K texture maps for v1, architecture supports swapping to 8K later
- Bloom/glow post-processing for Sun and atmospheric bodies
- Starfield skybox from real star catalog data
- WebGL is the baseline (all features work); WebGPU adds enhancements when available (compute-based particle effects, higher instance counts for Phase 3 asteroid belt)

**Phase 2+:**
- LOD system: high-detail meshes when close, simplified at distance (unnecessary at 9 bodies, needed at 30+)
- Instanced rendering for asteroid belt region (Phase 3)

### 2. Orbital Mechanics
- `astronomy-engine` (MIT, JS, client-side) for real-time ephemeris
- Positions updated at fixed intervals (not every frame — ephemeris is smooth enough at 1Hz)
- v1: real-time orbits only (planets move at actual orbital speed)
- Moon positions relative to parent planet
- **Phase 2:** Time controls (accelerated, paused, scrub) — deferred because user controls position, not time

### 3. Audio Engine (Web Audio API)
- AudioContext with per-stem GainNode graph
- Distance-based gain curves per body. Example config:
  ```json
  { "body": "Jupiter", "curve": "logarithmic", "audibilityRadius": 5e8, "maxGain": 0.8, "stems": 3 }
  { "body": "Ceres", "curve": "logarithmic", "audibilityRadius": 1e7, "maxGain": 0.5, "stems": 0, "pool": "dwarf-planet" }
  ```
  Bodies with `stems: 0` use a shared audio pool keyed by category. Bodies with no pool and no stems are visual-only (silent).
  A shared pool contains N stems (e.g., 2-3). When a body using a pool enters audibility range, it draws stems from the pool (round-robin). Two bodies sharing a pool may play the same stems simultaneously — acceptable since they'll be at different distances/gains.
- Stem lifecycle: pre-fetch when moving toward a body (before entering audibility radius), decode, connect to gain node, suspend when leaving. Crossfade completes within 2-5 seconds of travel at default speed.
- Deep space procedural layer: Web Audio API oscillator/filter chain (low-frequency drone + filtered noise) that fades UP inversely to all body proximities
- Distance calculations: simple sort of all bodies each frame (trivial at 50 bodies, no spatial indexing needed)
- Mobile: same Web Audio API via Capacitor WebView
- **Error handling:** If a stem fails to decode, skip it and keep other stems + drone playing. Log the failure for debugging.
- **Performance budget:** Decode latency < 500ms per stem on mid-range mobile. Maximum concurrent decoded stems: 10.
- **Units:** All distances in kilometers (matching astronomy-engine output).

### 4. Navigation System
- **Free flight:** WASD + mouse/drag (desktop), virtual joystick + gyro (mobile). Default speed: ~1 AU/min.
- **Smooth journey:** Predefined camera paths (Catmull-Rom splines through waypoints), speed controlled by user. Variable speed, arriving at each waypoint in ~30s.
- **Focus travel:** Select body from list/map, camera bezier-curves to destination. Duration: 10-20s depending on distance.
- All navigation modes are keyboard-accessible from Phase 1 (arrow keys as WASD alternative, Tab to cycle body list for focus travel, Space to toggle journey mode).
- Camera always has subtle drift/float for zero-gravity feel (respects `prefers-reduced-motion`).
- Speed scaling: auto-adjust based on proximity (slow near bodies, fast in open space)

### 5. Asset Pipeline
- Textures: progressive loading (low-res placeholder -> full-res swap)
- Audio stems: AAC (M4A) as primary format for universal support (including iOS/Safari); OGG as optional higher-quality alternative for browsers that support it. Serve appropriate format via feature detection.
- Self-hosted web server serves all assets
- Mobile: small bundle < 50MB (app shell + low-res textures), streams high-res + audio from web server
- CDN optional for production scaling

### 6. Mobile Shell (Capacitor)
- Ionic Capacitor wrapping the same web app
- Native plugins for: audio session management (iOS background audio), haptic feedback, gyroscope
- Small APK/IPA: < 50MB bundled (canonical target), streams everything else

### 7. UI/HUD (toggleable via settings)
- Body name labels (HTML overlay or Three.js sprites)
- Distance readout to nearest/selected body
- Mini solar system map for orientation
- Info panels (body stats)
- Audio stem visualization (which stems active, gain levels)
- Settings panel: toggle all HUD elements, audio quality, texture quality

## Resilience & Offline Behavior

- **Offline mobile:** Minimum cached audio set (deep space drone + stems for nearest 3 bodies at last session end). Graceful degradation when streaming assets fail — show low-res placeholder textures, keep cached audio playing.
- **State persistence:** Session state (camera position, active stems, settings) persisted to IndexedDB on web, Capacitor Preferences on mobile.
- **"Downloaded for offline" mode:** Users can opt to download all stems for a region (e.g., "Inner planets pack") for offline sessions.
- **Render fallback chain:** WebGPU -> WebGL -> "Your browser doesn't support 3D rendering" message with link to supported browsers.
- **Ephemeris error:** If `astronomy-engine` returns invalid positions, fall back to static Keplerian approximation positions (hardcoded for the 9 Phase 1 bodies).

## Accessibility (Phase 2)

- Keyboard-only navigation (arrow keys + speed controls)
- Reduced-motion preference: disable camera drift, slow transitions
- High-contrast mode for labels/HUD
- Screen reader support for info panels (ARIA labels on HUD elements)
- Colorblind-safe HUD colors

## Success Criteria

1. **Core mechanic validation:** Navigate from Jupiter to Saturn in the web version and feel the audio shift in a way that's emotionally compelling. Measurable proxy: at least 3 distinct audio zones perceptible in a Jupiter-to-Saturn traversal, with crossfades completing within 2-5 seconds at default speed.
2. **Performance:** 60fps on mid-range desktop, 30fps on mid-range mobile (Capacitor)
3. **Audio seamlessness:** No audible pops, clicks, or jarring transitions when stems crossfade
4. **Mobile parity:** Same core experience on iOS/Android via Capacitor, with touch navigation
5. **Scale test:** 50 bodies rendered simultaneously with audio for nearest 5-10 active without performance degradation
6. **Self-hostable:** Web version deploys to any static hosting or simple Node server

## Distribution Plan

- **Web (flagship):** Self-hostable static build (no vendor lock-in). Can deploy to any static host, own server, or managed platforms (Vercel/Netlify/Cloudflare Pages) as convenience.
- **Mobile:** Capacitor builds submitted to iOS App Store and Google Play Store. Small bundle (~30-50MB), streams textures and audio. Minimum target: Android 10+ (Snapdragon 665+), iOS 15+.
- **CI/CD (Phase 1):** GitHub Actions for web builds only. Automated deployment to web hosting on merge to main.
- **CI/CD (Phase 2):** Add Capacitor iOS/Android builds (code signing, provisioning profiles) when mobile is ready for store submission.

Sit down with your audio partner and compose 3 stems for Jupiter and 2 stems for Saturn. Load them in a basic Web Audio API test page.
